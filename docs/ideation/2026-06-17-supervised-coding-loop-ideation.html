Grounding Context

The headline: this is a wiring problem, not a build problem. The four systems you need already exist and are proven separately. The supervised loop is the integration that nobody on the market has shipped — the literature has a generic supervisor (OpenAI Auto-review) and a generic spec-driver (Kiro, OpenHands), but no one wires a persona-oracle + spec + an independent referee into one loop.

The request maps cleanly onto the 2026 governance vocabulary: move from human-in-the-loop (approve every step — today) to human-in-command (you set the rules + the northstar; the loop executes within them and escalates by exception). The prior art and its hazards:

  • OpenAI Auto-review — a separate reviewer agent grades the coding agent's escalation requests: ~200× fewer human interventions, 99.1% auto-approve. The production proof a supervisor works — but its policy is generic, not your taste.
  • PersonaTwin names the two killer risks: preference hallucination (confident fabricated preference on a novel fork) and persona drift (the twin wanders over a long session). Both are designed against below.
  • Microsoft red-team (Jun 2026): human-in-the-loop bypass via consent fatigue, memory poisoning of the persona, and silent Goodhart spec failures.

01

Fork-Card decision channel — the agent emits a decision, it never pauses

decision-interception confidence: High complexity: High keystone
What

Today Claude Code pauses and waits for you at every "which option?" fork. Wrap claude -p with a decision-emission contract: when the agent would ask, it instead writes a typed ForkCard{options[], spec_clause, reversibility, blast_radius} to a queue the supervisor consumes — no terminal pause. The agent keeps building the reversible default; only a reversal of that default triggers the loop's existing rollback. The fork becomes a first-class, gradable build artifact, exactly as the engine already treats a target.

Basis
direct — adapters/compound_engineering.py (ClaudeCodeRefiner already drives the agent headlessly; the only missing primitive is a decision channel) + controller.py regression rollback for the reversal case.
Why it matters

This is the single unblock that converts "human babysits" into "supervisor answers." Every other idea depends on this surface existing. Reframing the fork as a non-blocking emitted artifact also means the build never stalls on an oracle round-trip.

Watch out

Claude Code doesn't natively emit typed Fork-Cards — you shape this via the /ce-work prompt convention (and/or pause-detection). This is the hard part of the whole project; prototype the contract first.

Meeting test

"Should a decision be a blocking question or a reversible default we can un-pick?" is the first architecture argument the team must have.

02

The twin SCORES the options + cite-or-escalate — it never authors a free-text answer

persona-oracle confidence: High complexity: Medium
What

Break the assumption that the oracle replies in prose. super-u's Digital Twin already scores toward/sideways/away a Vision Kernel — so point it at the Fork-Card's enumerated options and have it rank them with a per-option alignment delta. The supervisor auto-picks argmax only if (a) the margin clears a threshold AND (b) the choice cites a kernel.yaml clause or a dreammaketrue person-map node. No citation, weak margin, or all-sideways → escalate. Require two grounding sources to agree, so a single poisoned source can't carry a decision.

Basis
direct — super-u toward/sideways/away scoring (kernel.yaml) + dreammaketrue refuse-to-hallucinate person-maps.
external — PersonaTwin (arXiv 2508.10906) names "preference hallucination" as the exact failure this prevents.
Why it matters

A scorer over a closed option set structurally cannot fabricate a preference for a case it has no grounding on — it returns low confidence and escalates. A generative oracle confidently invents one. The score+citation tuple is also the auditable artifact that makes "the supervisor decided X" reviewable later.

Watch out

The twin scores words/actions today; scoring build-decision options is a real (but small) extension — frame each option as a scorable item.

Meeting test

"Does the twin generate the answer, or just grade the choices?" decides whether the whole loop is safe.

03

oracle ≠ checker ≠ maker — a three-body separation, asserted fail-closed

spec-northstar-reasoning confidence: High complexity: Low cheapest soundness win
What

Name the boundary the design only implies. integrity.py already enforces maker≠checker via assert_maker_distinct_from_checker. The supervisor adds a third role — the persona oracle that answers forks. Add assert_oracle_distinct_from_checker: the twin that resolves a decision must never be the referee (CLIJudge) that grades the build toward the northstar. Three roles, three distinct objects, asserted before any run.

Basis
direct — integrity.py assert_maker_distinct_from_checker ("a maker cannot referee its own work, R10") + confirm_convergence as the sole shippability authority.
Why it matters

Without it, a persona-aligned-but-wrong build scores well because the same persona graded it — Goodhart through the back door, the exact sycophancy failure the red-team flags. This is the one-line invariant that makes persona-driving sound, and it reuses an assertion the repo already ships.

Meeting test

A senior engineer will immediately ask "what stops the twin from grading its own decisions?" This is the enforceable answer.

04

Spend the human only where it matters — reversibility-gating + an escalation budget + a merge-readiness pack

human-at-the-edges confidence: High complexity: Medium
What

Three cousin mechanisms that all protect your scarce attention. (a) Reversibility-gating: auto-resolve trivially-reversible forks (naming, layout, a lib behind a wrapper) even at medium confidence; route irreversible ones (schema migration, public API shape, auth boundary) to you regardless of persona confidence. (b) Escalation budget: a hard "N questions per build" cap forces the supervisor to triage by expected regret = uncertainty × irreversibility × blast-radius. (c) Merge-readiness pack: on convergence, the human doesn't review a green checkmark — they review a structured pack (every Fork-Card, option chosen, grounding citation, alignment margin, spec clause served, which forks were escalated, and a "least-confident decisions" shortlist), built by extending describe_gate_reason.

Basis
direct — integrity.py VerificationGate / confirm_convergence / describe_gate_reason (legible firing reason).
external — OpenAI Auto-review's ~200× fewer interventions; Microsoft red-team "consent fatigue / rubber-stamp" failure mode; ALMAS escalate-with-action-history.
Why it matters

The whole value prop ("you only review the end") collapses if the end-review is a rubber stamp. Scarcity makes each human touch deliberate; the least-confident shortlist points your eye exactly where judgment was thinnest. This is what keeps you genuinely in command, not nominally in the loop.

Meeting test

"When I approve the final product, what am I actually looking at?" — a decision trail, not a diff dump.

05

Decision Ledger — answer each fork once, replay it forever

convergence-to-end-result confidence: High complexity: Medium
What

Persist every resolved fork as {fork_fingerprint → option, citation, score, outcome} in a project-local ledger. Before escalating or re-asking, the supervisor checks the ledger; a matching fingerprint replays the prior decision and its grounding. This kills the "context lost between sessions / re-deciding the same thing" pain. The natural extension: when the twin has zero grounding on a reversible fork, don't escalate — build both branches in parallel worktrees (the fleet already does this) and let the referee break the tie, then bank the answer in the ledger.

Basis
direct — refactor_brief.py build_refactor_brief(upstream_outcomes=...) already threads prior outcomes into briefs; the ledger is the persistent, fingerprinted version. The race-both extension reuses coordinator.py worktree waves.
external — ALMAS summarized-action-history.
Why it matters

Decisions become a compounding asset instead of evaporating per session. The second build of a similar feature inherits the first build's judgment — and "no grounding" turns from a dead-end escalation into free, refereed exploration.

Meeting test

"Why did the agent ask me the identical question it asked last Tuesday?" It shouldn't.

06

Living spec — the northstar is a runnable referee, and every resolved fork patches it

spec-northstar-reasoning confidence: Medium complexity: Medium
What

Break "the spec is a static doc." Treat OBJECTIVES/SPEC as a constitution the CLIJudge grades against (Constitution > Spec > Tasks), AND make each oracle-resolved Fork-Card append a GIVEN/WHEN/THEN clause back to the spec. The decision the twin made on iteration 3 becomes a referee-checkable acceptance criterion on iteration 4 — the spec densifies exactly at the ambiguity points that forced a fork. When a fork has weak persona grounding but clear spec relevance, the supervisor resolves it against the governing clause, not vibes.

Basis
direct — adapters/judge.py CLIJudge grades toward a suite/spec; docs/solutions/outer-loop-non-gaps.md "quality only from the referee."
external — Spec-Driven Development / Kiro; Fowler's caveat that the spec must judge output after the fact.
Why it matters

Silent Goodhart failures happen because the spec is sparse where the real decisions live. Auto-densifying the spec at decision points is the highest-leverage place to add referee coverage — and it routes forks by type: persona answers "what do I want," spec answers "what does the project require."

Meeting test

"The spec writes itself at the decision points" is a genuinely new mechanism, not a config knob.

07

Golden-label flywheel + drift tripwire — the loop gets more unattended the more you use it

persona-oracle confidence: High complexity: Medium the 10X payoff
What

Flywheel: when you overturn a supervisor decision at end-review, that correction is written back as a golden label on a build decision (and, if it reveals a missing principle, proposed as a Vision-Kernel / spec edit). super-u's golden-label loop already calibrates the twin against your own corrected labels — so each review tightens the oracle and shrinks future escalations, OpenAI-Auto-review-style. Tripwire: on long unattended runs the twin can drift, so every K resolved forks re-score a frozen held-out slice of golden.jsonl; if alignment decays past a threshold, halt-and-escalate with the action history rather than keep auto-resolving on a drifted oracle.

Basis
direct — super-u golden-label loop (golden.jsonl, calibration / anti-drift); controller.py regression-rollback is the structural analog the tripwire reuses.
external — OpenAI Auto-review (200× fewer interventions as the policy learns); PersonaTwin persona-drift; Microsoft memory-poisoning.
Why it matters

This is the actual 10X compounding mechanism: it turns your scarcest output — taste — into a durable, measurable asset, so review #2 is cheaper than review #1. The tripwire is what makes a multi-hour unattended run allowed at all: a cheap canary against frozen truth catches a twin that's quietly gone sycophantic.

Meeting test

"Does correcting it once make it stop asking, or do I correct it forever?" — and "how would you even know the persona drifted during a 6-hour run?"

The honest bottom line

10X framing

You asked whether loop-anything can build a supervisor that drives the coding agent to the end. The answer is yes, and you already own all four parts — the loop engine, the Digital Twin, the person-map oracle, and the coding agent. Nobody on the market has wired persona + spec + an independent referee into one loop. The minimum viable version is just three ideas: #1 (emit forks instead of pausing), #2 (twin scores + cites or escalates), #3 (oracle ≠ checker, asserted). #7 is what makes it compound — each end-review you do makes the next build need fewer.
